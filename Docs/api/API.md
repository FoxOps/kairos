# API

For a complete machine-readable spec (schemas, error codes), see
[`openapi.yaml`](openapi.yaml) — importable into Swagger UI, Postman,
Insomnia, etc.

## Important: this application is not a REST API

Almost all of Kairos's routes render server-side HTML pages
(Jinja2), not JSON — it's a traditional web application, not an SPA
consuming an API. Only the endpoints listed below return JSON or ICS
content. For every other route (creating a user, bulk-adding a shift
from the admin form, etc.), see the
[Administrator Guide](../guides/ADMIN_GUIDE.md) and the
[User Guide](../guides/USER_GUIDE.md).

## Authentication

- **Session**: Flask-Login session cookie, created by `POST /login`
  (email/password) or the OIDC flow (`GET /oidc/login` →
  `GET /oidc/callback`).
- **CSRF**: every write request (POST/PUT/PATCH/DELETE) must include
  the `X-CSRFToken` header, whose value is read from
  `<meta name="csrf-token">` on server-rendered HTML pages.
  Without it: `400 Bad Request`. See
  [`architecture/SEQUENCE_DIAGRAMS.md`](../architecture/SEQUENCE_DIAGRAMS.md#updating-a-shift-via-drag-and-drop-json-api--csrf).
- **ICS token** (`export/*` only): bearer token in the `?token=...`
  query parameter, generated by a user via
  `POST /profile/ics-token`. Designed for external calendar
  subscriptions (Google Calendar, Outlook) that don't carry a cookie.

## JSON endpoints

### Schedule (`/api/shifts`, `/api/users`, `/api/shift-types`)

| Method | Route | Auth | Description |
|---|---|---|---|
| GET | `/api/shifts` | logged in | Calendar events (shifts **+ on-calls + leaves**, despite the name) over ±180 days, FullCalendar format |
| POST | `/api/shifts` | admin | Creates a shift (`userId`, `shiftTypeId`, `start`, `end?`) |
| PATCH/PUT | `/api/shifts/<id>` | admin | Moves/resizes (`start`, `end?` — duration kept if `end` is absent) |
| DELETE | `/api/shifts/<id>` | admin | Deletes |
| GET | `/api/users` | logged in | Visible users (admin: all; regular user: themselves) |
| GET | `/api/shift-types` | logged in | List of shift types |

Validation rules shared by POST/PATCH `/api/shifts`:
- Rejected (`400`) if the date falls on a Saturday/Sunday.
- Rejected (`400`) if the user already has a shift that day (conflict).

### On-call (`/api/oncall/<id>`)

| Method | Route | Auth | Description |
|---|---|---|---|
| PATCH/PUT | `/api/oncall/<id>` | admin | Moves (`start` must fall on a **Friday**, `end?`) |
| DELETE | `/api/oncall/<id>` | admin | Deletes |

There is **no** `POST /api/oncall` — creation only happens
via the `/oncall/add` form (server-rendered), not through the JSON API.

### Leave (`/api/leave/<id>`)

| Method | Route | Auth | Description |
|---|---|---|---|
| PATCH/PUT | `/api/leave/<id>` | logged in | Modifies the dates (`start`, `end?`) — restricted to one's own leave unless admin |
| DELETE | `/api/leave/<id>` | logged in | Deletes — same restriction |

Unlike shifts and on-calls (admin-only), a regular user
can modify/delete **their own** leave via the API — a
`403` is returned if they try to act on another user's leave.

### ICS export (`/export/*`)

| Method | Route | Auth | Description |
|---|---|---|---|
| GET | `/export/shifts?scope=my\|all&token=...` | session or token | `.ics` export of shifts |
| GET | `/export/oncall?scope=my\|all&token=...` | session or token | `.ics` export of on-calls |
| GET | `/export/leaves?scope=my\|all&token=...` | session or token | `.ics` export of leave |

Default `scope`: `all`. `scope=my` filters on the resolved user
(session or token). Response `401` if neither a valid session nor a
valid token is present (or a redirect to `/login` for a regular browser
client).

## Response formats

All write JSON responses (POST/PATCH/DELETE) follow the same
format:

```json
// Success
{ "success": true, "message": "...", "shift": { /* created/modified object, if relevant */ } }

// Error
{ "success": false, "error": "error message" }
```

## HTTP status codes used

| Code | Meaning in this context |
|---|---|
| 200 | Success |
| 400 | Validation failed (dates, conflict, weekend, missing/invalid JSON, missing/invalid CSRF) |
| 401 | Not authenticated (HTML routes: redirect to `/login` instead) |
| 403 | Authenticated but not authorized (non-admin on an admin route, or another user's leave) |
| 404 | Resource not found |
| 500 | Unexpected server error |

## Referenced data models

See [`architecture/ERD.md`](../architecture/ERD.md) for the complete
schema. Fields worth remembering for the API:

- A JSON **shift** object exposes `id`, `title`, `start`, `end`,
  `className`, `userId`, `shiftTypeId` — not the `Shift` model's
  columns directly.
- The `Leave` model **has no `reason` field** despite what the
  old version of this document described.
- `User.to_dict()` always excludes `password_hash` and `ics_token` — these
  fields are never present in a JSON response, including
  `/api/users`.

## Public API v1 (service accounts)

**A second API surface, distinct from everything above** — do not
confuse the two authentication mechanisms. Designed for
third-party integrations (Zapier, external scripts, reporting
tools), not for the application's own frontend (which continues to
use the internal `/api/*` API above, session cookie).

- **Base URL**: `/api/v1/*` — a prefix deliberately distinct from
  `/api/*` (internal API) to avoid any collision.
- **Authentication**: `Authorization: Bearer <token>` header,
  never a session cookie. The token is generated from
  `/admin/service-accounts` (admin-only) and shown in
  clear text **only once**, at creation or regeneration — it
  is never shown again afterward (only a truncated prefix stays
  visible in the list, for identification). A missing, invalid,
  revoked, or expired token returns `401` with a JSON body
  `{"message": "..."}`  — never an HTML redirect, unlike the
  classic `/api/*`/HTML routes.
- **CSRF**: not applicable — this API never accepts a cookie, so
  no `X-CSRFToken` is required.
- **Rate limiting**: keyed by service account identity (not by IP),
  `60 requests/minute, 1000/day` by default.
- **v1 scope**: **read-only**. `GET /api/v1/shifts[/<id>]`,
  `GET /api/v1/oncall[/<id>]`, `GET /api/v1/leave[/<id>]`,
  `GET /api/v1/users[/<id>]`, `GET /api/v1/shift-types`. Lists are
  paginated (`?page=`, `?per_page=`), with the same default
  settings as the admin UI (`items_per_page`/`max_per_page`, `/admin/settings`).
  Write support is not planned for v1 — a future enhancement if a
  real need arises, not an oversight.
- **Machine-readable documentation**: `GET /api/v1/openapi.json`, an
  OpenAPI 3.0.3 spec **automatically generated** from the
  marshmallow schemas (`app/api/schemas/`) on every startup — unlike
  [`openapi.yaml`](openapi.yaml) above (internal API), it cannot
  drift out of sync with the real code. No interactive Swagger/Redoc UI
  is served by the application (the site's strict CSP doesn't
  whitelist the CDN these UIs need by default) — import `openapi.json`
  into an external Swagger UI/Postman/Insomnia to explore it visually.
